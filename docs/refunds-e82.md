# E82: Stripe + Mercado Pago — возвраты (admin + organizer)

Реализована архитектура возвратов без фиктивных «успешно»:

- сначала создаётся запись в `payment_transactions` с типом `refund` и статусом `pending`;
- затем система пытается отправить возврат в провайдера, только если конфигурация доступна;
- при отсутствии ключей запрос остаётся в `pending` для ручной обработки.

## Новые API

### `POST /api/admin/payments/refund`

Создаёт запрос на возврат из админки по `bookingId`, валидирует оплату и запускает попытку возврата:

- Stripe: при наличии `STRIPE_SECRET_KEY`;
- Mercado Pago: при наличии `MERCADOPAGO_ACCESS_TOKEN` и `MERCADOPAGO_REFUNDS_ENABLED=true`.

Если провайдер не настроен или вернул ошибку, запрос всё равно сохраняется в журнале как `pending`.

### `POST /api/organizer/payments/refund-request`

Позволяет организатору запросить возврат по своей заявке:

- проверяется роль организатора;
- проверяется владение бронированием (`organizerCanAccessBooking`);
- допускается только полностью оплаченная заявка (`paid`).

После создания запроса выполняется та же попытка отправки в провайдера.

## Обновления существующего API

### `GET /api/bookings/[id]/payment/refund`

Теперь возвращает:

- `pendingRefund` — ожидающий запрос;
- `latestRefund` — последний запрос на возврат любого статуса.

Это позволяет показывать туристу и организатору не только «ожидает», но и `completed`/`rejected`/`failed`.

## UI

## Админка

- В деталях транзакции `charge` добавлено действие «Создать запрос на возврат».
- В деталях `refund` со статусом `pending` добавлены кнопки «Одобрить возврат»/«Отклонить».
- Статусы отображаются как бейджи (`pending`, `processing`, `completed`, `failed`, `rejected`, `cancelled`).

## Организатор

- В карточке заявки кнопка «Запросить возврат» работает через `POST /api/organizer/payments/refund-request`.
- Кнопка доступна только для полностью оплаченной заявки.

## Турист

- На странице деталей бронирования отображается статус последнего запроса на возврат, если он существует.

## Примечание по данным

Новая миграция не требуется:

- таблица `payment_transactions` уже содержит `metadata jsonb`, куда записываются служебные данные попытки возврата.
