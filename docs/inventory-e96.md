# E96 — Инвентарь и доступность туров

## Что добавлено

- Таблица `tour_availability_slots` для хранения слотов доступности по датам:
  - `tour_id`, `date`, `capacity`, `booked_count`, `status`.
- Таблица `tour_waitlist_entries` для заявок в лист ожидания:
  - `tour_id`, `user_id`/`email`, `slot_date`, `status`, контактные поля.
- Публичный API:
  - `GET /api/tours/[slug]/availability` — доступность слотов для карточки тура и checkout.
  - `POST /api/tours/[slug]/waitlist` — вступление в лист ожидания.
- Проверка вместимости в checkout перед созданием бронирования:
  - в `POST /api/bookings` добавлена валидация/резервирование места в слоте.
- Организаторский API и UI:
  - `GET/PUT /api/organizer/tours/[id]/availability` для управления слотами.
  - в редакторе тура (блок групповых дат) добавлена синхронизация слотов в БД.

## Поведение при пустых таблицах

Если в `tour_availability_slots` нет строк, API возвращает фолбэк-слоты на основе `booking.groupDates` из канонического тура (seed/черновик).  
Это позволяет не ломать checkout и отображение дат до первой синхронизации инвентаря.

## Статусы слотов

- `open` — бронирование открыто.
- `sold_out` — мест не осталось.
- `closed` — дата закрыта вручную.

## Поток бронирования

1. Клиент отправляет заявку в `POST /api/bookings`.
2. Сервер проверяет слот на выбранную дату (`tour_id + start_date`).
3. При успехе временно резервирует места (`booked_count += guests`).
4. Создаёт запись бронирования.
5. Если вставка бронирования не удалась, резерв откатывается.

## Поток листа ожидания

1. Клиент отправляет `POST /api/tours/[slug]/waitlist`.
2. Сервер валидирует тур и дату слота.
3. Создаёт запись `tour_waitlist_entries` со статусом `waiting`.
4. Для обратной совместимости в UI оставлен мягкий фолбэк на localStorage при недоступном API.

## Примечание по правам

- Для таблиц включён RLS.
- Публичное чтение слотов доступно `anon/authenticated`.
- Публичная запись в лист ожидания разрешена только для создания статуса `waiting`.
- Полный доступ для фоновых/серверных операций — через `service_role`.
