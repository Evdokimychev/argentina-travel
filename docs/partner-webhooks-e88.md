# E88: Партнёрские вебхуки событий бронирования

## Что реализовано

- Добавлены таблицы `partner_webhooks` и `partner_webhook_deliveries`.
- Добавлен CRUD API для организатора: `/api/organizer/webhooks` и `/api/organizer/webhooks/[id]`.
- Добавлен тестовый ping: `/api/organizer/webhooks/[id]/test`.
- Подключён диспетчер событий:
  - `booking.created`
  - `booking.confirmed`
  - `booking.cancelled`
- Реализованы ретраи доставки: до 3 попыток с экспоненциальной задержкой.
- В кабинете организатора на странице интеграций добавлен интерфейс управления вебхуками.

## Таблицы

### `partner_webhooks`

- `organizer_id` — владелец интеграции
- `url` — URL приёмника
- `secret` — общий секрет для подписи `HMAC-SHA256`
- `events[]` — подписка на события
- `active` — активность вебхука

RLS:

- организатор видит и изменяет только свои вебхуки;
- `service_role` имеет полный доступ.

### `partner_webhook_deliveries`

- `webhook_id` — ссылка на вебхук
- `event` — тип события
- `payload` — JSON полезной нагрузки
- `status` — `pending | delivering | delivered | failed`
- `attempts` — количество попыток (`0..3`)

RLS:

- организатору доступен просмотр доставок своих вебхуков;
- `service_role` имеет полный доступ.

## Формат подписи

Каждый POST в вебхук отправляется с заголовками:

- `X-PVA-Event`
- `X-PVA-Signature`
- `X-PVA-Webhook-Id`
- `X-PVA-Delivery-Id`

Подпись:

- алгоритм: `HMAC-SHA256`
- строка: сырой JSON payload
- формат заголовка: `sha256=<hex>`

## Повторные попытки

Для каждой доставки:

1. попытка сразу;
2. при ошибке вторая попытка через `500ms`;
3. при ошибке третья попытка через `1000ms`.

При финальной неудаче доставка получает `status = failed`.

## UI организатора

На странице `/organizer/integrations` добавлены:

- форма добавления вебхука (`URL`, `секрет`, список событий);
- список подключённых вебхуков;
- кнопка тестовой отправки (ping);
- включение/отключение и удаление вебхука.

## Ограничения и безопасность

- Если URL не задан, исходящий вызов не выполняется.
- В тестовой отправке и реальных событиях используется один и тот же механизм подписи.
- Секрет не возвращается в API в явном виде (только маска).
