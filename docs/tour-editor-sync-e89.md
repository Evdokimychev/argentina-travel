# E89: Tour editor Supabase sync v2

## Цель

Уменьшить зависимость редактора тура от сценария `localStorage-only` и добавить управляемую синхронизацию черновика с Supabase для авторизованного организатора.

## Что реализовано

1. **Автосохранение в Supabase (debounce 3 секунды)**
   - В `OrganizerTourEditorView` автосейв срабатывает через `useDebouncedValue(..., 3000)`.
   - Локальный черновик сохраняется сразу в `localStorage`.
   - Отправка на сервер идёт через `PATCH /api/organizer/tours/[id]/draft`.
   - Для вызовов из редактора у `saveOrganizerTourDraft` включён флаг `skipRemoteSync`, чтобы исключить параллельный legacy-sync и гонки.

2. **Обнаружение конфликта**
   - На клиенте хранится `serverUpdatedAt` (последняя подтверждённая серверная версия).
   - `PATCH` принимает `expectedUpdatedAt` и возвращает `409`, если `server.updated_at > expectedUpdatedAt`.
   - В редакторе показывается диалог конфликта с кнопкой «Перезаписать сервер».
   - В шапке редактора отображается статус `Конфликт`.

3. **Офлайн-очередь с `localStorage`**
   - Если сеть недоступна или запрос временно неуспешен, черновик кладётся в очередь
     `argentina-travel-organizer-tour-draft-sync-queue-v2`.
   - Очередь автоматически пытается отправиться:
     - при открытии редактора;
     - при событии `online` (восстановление сети).

4. **Индикатор состояния в шапке**
   - Добавлен единый бейдж статуса:
     - `Сохранено`
     - `Сохранение…`
     - `Конфликт`
   - Тот же статус дублируется в мобильном/боковом блоке редактора.

5. **Новый API draft-sync**
   - `GET /api/organizer/tours/[id]/draft`
     - возвращает серверные `updatedAt` и текущий canonical tour.
   - `PATCH /api/organizer/tours/[id]/draft`
     - проверяет роль организатора и владение туром;
     - выполняет conflict-check по `updated_at`;
     - обновляет запись в таблице `tours`.

## Почему publish-flow не ломается

- Локальное сохранение черновика осталось прежним (включая существующую валидацию readiness перед публикацией).
- Публикация по-прежнему проходит через тот же поток `persistDraft`.
- Новая синхронизация не блокирует локальную работу: при офлайне данные не теряются и отправляются позже из очереди.
