# E94 — Server-side агрегация аналитики организатора

## Что реализовано

- Добавлен серверный агрегатор `src/lib/organizer/analytics-server.ts`.
- Добавлен API `GET /api/organizer/analytics?period=30d` с проверкой роли организатора.
- В тот же endpoint добавлена выгрузка CSV через параметр `format=csv`.
- `OrganizerAnalyticsView` переведён на серверный источник в режиме Supabase.
- Сохранён fallback на текущую клиентскую аналитику, когда remote-режим выключен.

## Источники данных

Агрегация строится по:

- `bookings` (фильтр `organizer_user_id`)
- `payment_transactions` (транзакции type=`charge` для найденных бронирований)
- `analytics_events` (воронка по `tour_view` и `booking_started` для туров организатора)

## Метрики API

Ответ `report` включает:

- `summary`: выручка, число бронирований, подтверждённые/оплаченные, средний чек, уникальные клиенты
- `funnel`: просмотры туров, старты бронирования, подтверждения, оплаты и три конверсии
- `topTours`: топ-10 туров по выручке с конверсией по каждому туру

## CSV

Кнопка «Скачать CSV» в кабинете организатора запрашивает:

- `GET /api/organizer/analytics?period=<period>&format=csv`

Файл содержит:

- блок основных KPI и воронки
- таблицу top tours (slug, бронирования, выручка, конверсия)

## Замечания по устойчивости

- Если в `analytics_events` мало данных, воронка не «падает»: `bookingStarts` и `tourViews`
  корректируются на основе фактических бронирований.
- Для выручки сначала используются успешные транзакции, а при их отсутствии — подтверждённые
  суммы из бронирований.
